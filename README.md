# Частина 1. Проєктування схеми графа

## Схема графа

### Вузли

```text
(User {userId, gender, age, occupation})
(Movie {movieId, title, year})
(Genre {name})
```

### Ребра

```text
(User)-[:RATED {rating, timestamp}]->(Movie)
(Movie)-[:HAS_GENRE]->(Genre)
```

## ASCII-діаграма

```text
+----------------------+
|        User          |
|----------------------|
| userId               |
| gender               |
| age                  |
| occupation           |
+----------------------+
           |
           | RATED
           | {rating, timestamp}
           v
+----------------------+
|        Movie         |
|----------------------|
| movieId              |
| title                |
| year                 |
+----------------------+
           |
           | HAS_GENRE
           v
+----------------------+
|        Genre         |
|----------------------|
| name                 |
+----------------------+
```

## Пояснення схеми

У побудованій моделі використовується три основні типи вузлів: `User`, `Movie` та `Genre`.

Вузол `User` представляє користувача системи та містить його основні характеристики: ідентифікатор, стать, вік і професію. Саме користувачі створюють оцінки фільмів, тому вони є однією з центральних сутностей графа.

Вузол `Movie` представляє фільм і є основним об'єктом рекомендаційної системи. З ним пов’язані оцінки користувачів та жанри.

Вузол `Genre` представляє жанр фільму. Один жанр може бути пов’язаний із великою кількістю фільмів, а один фільм може належати до декількох жанрів.

---

## Відповідь на питання 1

### Які сутності стали вузлами, а які — ребрами? Чому?

Вузлами були обрані:

- `User`
- `Movie`
- `Genre`

Ці сутності мають власний набір атрибутів і можуть існувати незалежно одна від одної.

Ребрами були обрані:

```text
(User)-[:RATED]->(Movie)
(Movie)-[:HAS_GENRE]->(Genre)
```

Зв’язок `RATED` відображає дію користувача щодо фільму та має власні властивості (`rating`, `timestamp`).

Зв’язок `HAS_GENRE` показує належність фільму до певного жанру.

Такий підхід відповідає рекомендаціям для графових баз даних: сутності стають вузлами, а взаємодії між ними — ребрами.

---

## Відповідь на питання 2

### Оцінка користувача за фільм — це ребро чи окремий вузол?

У даній роботі оцінка реалізована як ребро:

```text
(User)-[:RATED {rating, timestamp}]->(Movie)
```

### Переваги такого рішення

1. Менша кількість вузлів у графі.
2. Простіші запити.
3. Швидші обходи графа.
4. Природне відображення факту, що оцінка є взаємодією між користувачем і фільмом.

Наприклад:

```cypher
MATCH (u:User)-[r:RATED]->(m:Movie)
RETURN u.userId, m.title, r.rating
```

---

### Альтернативний підхід

Можна було б створити окремий вузол:

```text
(User)-[:GAVE]->(Rating)-[:FOR]->(Movie)
```

де:

```text
(Rating {
    rating,
    timestamp
})
```

Такий підхід був би доцільним, якщо оцінка мала б:

- текстовий відгук;
- історію змін;
- лайки;
- коментарі;
- додаткові зв’язки.

---

### Висновок

Для датасету MovieLens оцінка складається лише з:

- rating;
- timestamp.

Тому окремий вузол `Rating` лише ускладнив би граф без додаткової користі. Використання ребра `RATED` є більш природним та ефективним рішенням.

---

## Відповідь на питання 3

### Чому жанри краще зберігати як окремі вузли?

Можливий альтернативний варіант:

```text
(Movie {
    title,
    genres: ["Action", "Thriller"]
})
```

Однак такий підхід має ряд недоліків.

### Недоліки зберігання жанрів у властивості

Складніше виконувати запити:

```cypher
MATCH (m:Movie)
WHERE "Thriller" IN m.genres
RETURN m
```

Неможливо будувати графові обходи через жанри:

```text
User -> Movie -> Genre
```

Складніше аналізувати популярність жанрів.

Неможливо легко знаходити зв’язки між жанрами та користувачами.

---

### Переваги окремих вузлів Genre

Запити стають більш природними:

```cypher
MATCH (m:Movie)-[:HAS_GENRE]->(g:Genre {name: "Thriller"})
RETURN m.title
```

Легко будувати статистику:

```cypher
MATCH (:Movie)-[:HAS_GENRE]->(g:Genre)
RETURN g.name, count(*)
```

Можна визначати:

- найпопулярніші жанри;
- жанрові кластери;
- жанрові супервузли;
- жанрові вподобання користувачів.

---

### Висновок

Винесення жанрів в окремі вузли дозволяє повніше використовувати переваги графової моделі та значно спрощує аналітичні запити. Саме тому в даній роботі жанри реалізовані через вузли `Genre` та зв’язки `HAS_GENRE`.



# Частина 2. Завантаження даних

## Підготовка CSV-файлів

Початкові файли MovieLens мають формат `.dat`, роздільник `::` та кодування `latin-1`.

Для завантаження в Neo4j дані були конвертовані у CSV-файли з кодуванням `utf-8`:

```text
movies.dat  -> import/movies.csv
ratings.dat -> import/ratings.csv
users.dat   -> import/users.csv
```

Для цього був використаний скрипт `convert.py`.

## Завантаження вузлів User

```cypher
LOAD CSV WITH HEADERS
FROM 'file:///users.csv' AS row

MERGE (u:User {userId: toInteger(row.userId)})
SET
    u.gender = row.gender,
    u.age = toInteger(row.age),
    u.occupation = toInteger(row.occupation);
```

Цей запит створює вузли `User`.

`MERGE` використовується замість `CREATE`, щоб уникнути дублювання користувачів при повторному запуску скрипту.

## Завантаження вузлів Movie

```cypher
LOAD CSV WITH HEADERS
FROM 'file:///movies.csv' AS row

MERGE (m:Movie {movieId: toInteger(row.movieId)})
SET
    m.title = row.title;
```

Цей запит створює вузли `Movie`.

Кожен фільм має унікальний `movieId`, тому саме це поле використовується для пошуку або створення вузла.

## Завантаження жанрів і зв'язків HAS_GENRE

```cypher
LOAD CSV WITH HEADERS
FROM 'file:///movies.csv' AS row

MATCH (m:Movie {movieId: toInteger(row.movieId)})

WITH m, split(row.genres, '|') AS genres

UNWIND genres AS genreName

MERGE (g:Genre {name: genreName})

MERGE (m)-[:HAS_GENRE]->(g);
```

Жанри у MovieLens зберігаються одним рядком, наприклад:

```text
Action|Adventure|Sci-Fi
```

Функція `split(row.genres, '|')` розбиває цей рядок на список жанрів.

`UNWIND` перетворює список на окремі рядки, після чого для кожного жанру створюється вузол `Genre`.

## Індекси

```cypher
CREATE INDEX user_id_index IF NOT EXISTS
FOR (u:User)
ON (u.userId);

CREATE INDEX movie_id_index IF NOT EXISTS
FOR (m:Movie)
ON (m.movieId);

CREATE INDEX genre_name_index IF NOT EXISTS
FOR (g:Genre)
ON (g.name);
```

Індекси створені для полів, які часто використовуються в `MATCH` та `MERGE`.

Особливо важливі індекси для `User.userId` та `Movie.movieId`, тому що саме ці поля використовуються під час створення великої кількості ребер `RATED`.

## Завантаження ребер RATED

```cypher
CALL apoc.periodic.iterate(
    "
    LOAD CSV WITH HEADERS
    FROM 'file:///ratings.csv' AS row
    RETURN row
    ",
    "
    MATCH (u:User {userId: toInteger(row.userId)})
    MATCH (m:Movie {movieId: toInteger(row.movieId)})
    MERGE (u)-[r:RATED]->(m)
    SET
        r.rating = toFloat(row.rating),
        r.timestamp = toInteger(row.timestamp)
    ",
    {
        batchSize: 10000,
        parallel: false
    }
);
```

Файл `ratings.csv` містить понад один мільйон оцінок, тому завантажувати всі ребра однією транзакцією небезпечно.

Для цього використано `apoc.periodic.iterate`, який розбиває завантаження на батчі.

Параметр:

```cypher
batchSize: 10000
```

означає, що Neo4j обробляє дані частинами по 10 000 рядків.

Параметр:

```cypher
parallel: false
```

використовується для безпечного послідовного завантаження без конфліктів між потоками.

## Перевірка результатів

Після завантаження були отримані такі результати:

```text
Users:      6040
Movies:     3883
Genres:     18
HAS_GENRE:  6408
RATED:      1000209
```

Це підтверджує, що дані MovieLens були успішно завантажені в Neo4j.



# Частина 3. Запити
## Запит 1. Фільми жанру Thriller із середнім рейтингом вище 4.0

```cypher
MATCH (m:Movie)-[:HAS_GENRE]->(:Genre {name: "Thriller"})
MATCH (m)<-[r:RATED]-(:User)
WITH
    m,
    avg(r.rating) AS avgRating,
    count(r) AS ratingCount
WHERE avgRating > 4.0
RETURN
    m.movieId,
    m.title,
    round(avgRating * 100) / 100 AS avgRating,
    ratingCount
ORDER BY avgRating DESC, ratingCount DESC
LIMIT 10;
```

### Пояснення

Запит знаходить усі фільми жанру Thriller та збирає всі оцінки користувачів для цих фільмів.

Для кожного фільму обчислюється:

- середній рейтинг (`avg(r.rating)`);
- кількість оцінок (`count(r)`).

Після цього залишаються лише фільми із середнім рейтингом вище 4.0.

Результат сортується за середнім рейтингом та кількістю оцінок.

#### Результат

Найвищі оцінки отримали:

- The Usual Suspects (1995)
- Rear Window (1954)
- The Sixth Sense (1999)
- The Silence of the Lambs (1991)
- The Matrix (1999)

Отримані результати відповідають очікуванням, оскільки ці фільми традиційно вважаються одними з найкращих представників жанру Thriller.

## Запит 2. Користувачі, які поставили оцінку 5 більш ніж 50 фільмам

```cypher
MATCH (u:User)-[r:RATED]->(:Movie)
WHERE r.rating = 5

WITH
    u,
    count(r) AS fiveRatings

WHERE fiveRatings > 50

RETURN
    u.userId,
    u.gender,
    u.age,
    fiveRatings

ORDER BY fiveRatings DESC;
```

### Пояснення

Запит знаходить усі оцінки зі значенням 5.

Після цього для кожного користувача підраховується кількість таких оцінок.

З результату відбираються лише користувачі, які поставили максимальну оцінку більш ніж 50 фільмам.

#### Результат

Найактивніший користувач поставив оцінку 5 для 571 фільму.
Отримані результати дозволяють виявити користувачів, які дуже часто залишають максимальні оцінки та можуть суттєво впливати на статистику популярності фільмів.


## Запит 3. Фільми, які два користувачі оцінили високо

```cypher
MATCH (u1:User {userId: 1})-[r1:RATED]->(m:Movie)
MATCH (u2:User {userId: 2})-[r2:RATED]->(m)

WHERE r1.rating >= 4
  AND r2.rating >= 4

RETURN
    m.movieId,
    m.title,
    r1.rating AS user1_rating,
    r2.rating AS user2_rating

ORDER BY m.title;
```

### Пояснення

Запит знаходить фільми, які були оцінені двома конкретними користувачами.

Використовується спільний вузол `Movie`, через який пов'язуються два користувачі.

Додаткова умова:

```cypher
r1.rating >= 4
AND r2.rating >= 4
```

залишає лише ті фільми, які сподобались обом користувачам.

### Результат

Було знайдено 6 фільмів, які користувачі з `userId=1` та `userId=2` оцінили рейтингом не нижче 4.

Серед них:

- One Flew Over the Cuckoo's Nest
- Saving Private Ryan
- To Kill a Mockingbird
- Dead Poets Society

Цей запит демонструє пошук спільних вподобань між користувачами та є базою для побудови рекомендаційних систем.


## Запит 4. Жанри з найвищими середніми оцінками

```cypher
MATCH (g:Genre)<-[:HAS_GENRE]-(m:Movie)<-[r:RATED]-(:User)

WITH
    g,
    avg(r.rating) AS avgRating,
    count(r) AS totalRatings

WHERE totalRatings > 1000

RETURN
    g.name AS genre,
    round(avgRating * 100) / 100 AS avgRating,
    totalRatings

ORDER BY avgRating DESC;
```

### Пояснення

Запит проходить по графу:

```text
Genre <- Movie <- RATED <- User
```

та збирає всі оцінки для кожного жанру.

Для кожного жанру обчислюється:

- середній рейтинг;
- загальна кількість оцінок.

Для підвищення достовірності результатів враховуються лише жанри, які мають більше 1000 оцінок.

### Результат

Найвищий середній рейтинг отримали:

- Film-Noir (4.08)
- Documentary (3.93)
- War (3.89)

Отримані результати показують, що жанри Film-Noir та Documentary отримують високі оцінки від користувачів, хоча можуть поступатися іншим жанрам за популярністю.

## Запит 5. Рекомендація фільмів на основі схожих користувачів

```cypher
MATCH (target:User {userId: 1})-[r1:RATED]->(m:Movie)<-[r2:RATED]-(other:User)

WHERE r1.rating >= 4
  AND r2.rating >= 4
  AND target <> other

WITH target, other, count(m) AS commonLikes

WHERE commonLikes >= 5

MATCH (other)-[r:RATED]->(recommended:Movie)

WHERE r.rating >= 4

AND NOT EXISTS {
    MATCH (target)-[:RATED]->(recommended)
}

WITH
    recommended,
    count(other) AS similarUsers,
    avg(r.rating) AS avgRating

RETURN
    recommended.movieId,
    recommended.title,
    similarUsers,
    round(avgRating * 100) / 100 AS avgRating

ORDER BY similarUsers DESC, avgRating DESC
LIMIT 20;
```

### Пояснення

Запит реалізує спрощений алгоритм collaborative filtering.

Спочатку знаходяться користувачі, які високо оцінили ті самі фільми, що й цільовий користувач.

Після цього для кожного такого користувача визначаються інші фільми, які він також високо оцінив.

З рекомендацій виключаються фільми, які цільовий користувач уже переглянув або оцінив.

Результати сортуються за кількістю схожих користувачів та середнім рейтингом.

### Результат

Серед рекомендованих фільмів були:

- Raiders of the Lost Ark
- The Shawshank Redemption

## Запит 6. Найкоротший шлях між двома користувачами

```cypher
MATCH p = shortestPath(
    (u1:User {userId: 1})-[*..10]-(u2:User {userId: 50})
)
RETURN p;
```

### Пояснення

Запит використовує функцію `shortestPath()` для пошуку найкоротшого шляху між двома користувачами.

Шлях будується через ребра `RATED` та вузли `Movie`.

Фактично алгоритм шукає мінімальний ланцюжок виду:

```text
User → Movie → User → Movie → User
```

### Результат

Для користувачів `userId=1` та `userId=50` був знайдений шлях довжини 2.

Це означає, що обидва користувачі оцінили один і той самий фільм.

### Відповідь на питання

Що означає довжина шляху?

Довжина шляху показує кількість ребер між двома користувачами.

Шлях довжини 2:

```text
User → Movie → User
```

означає, що користувачі мають один спільний фільм.

Шлях довжини 4:

```text
User → Movie → User → Movie → User
```

означає, що між користувачами існує один проміжний користувач.

Шлях довжини 6:

```text
User → Movie → User → Movie → User → Movie → User
```

означає, що між користувачами існує вже два проміжних користувачі.

Чим коротший шлях, тим ближчими можна вважати користувачів за їхніми кінематографічними вподобаннями.


# Частина 4. Виявлення супервузлів

## Запит для пошуку супервузлів

```cypher
MATCH (n)
WITH
    n,
    labels(n) AS labels,
    count { (n)--() } AS degree
RETURN
    labels,
    CASE
        WHEN "User" IN labels THEN toString(n.userId)
        WHEN "Movie" IN labels THEN n.title
        WHEN "Genre" IN labels THEN n.name
        ELSE toString(id(n))
    END AS nodeName,
    degree
ORDER BY degree DESC
LIMIT 20;
```

### Пояснення

Запит обчислює ступінь кожного вузла графа.

Ступінь вузла (`degree`) — це кількість зв'язків, які має вузол незалежно від напрямку ребер.

Для зручності відображення використовується конструкція `CASE`, яка показує:

- `userId` для вузлів User;
- назву фільму для Movie;
- назву жанру для Genre.

Результат сортується за кількістю зв'язків у спадному порядку.

### Чому запит через супервузол працює повільніше?

Індекс допомагає швидко знайти сам вузол.

Однак після знаходження вузла Neo4j повинен пройти всі його ребра.

Наприклад:

```text
American Beauty (1999)
3430 зв'язків
```

Щоб обробити цей вузол, Neo4j повинен переглянути всі 3430 ребер.

Для вузла з 50 зв'язками буде потрібно переглянути лише 50 ребер.

Саме тому запити, що проходять через супервузли, працюють повільніше навіть за наявності індексів.

## Відповідь на питання 3

### Яку стратегію оптимізації можна застосувати?

Для даного датасету доцільно використовувати підхід розбиття супервузлів (Node Splitting).

Особливо це стосується жанрових вузлів:

```text
Drama
Comedy
Action
```

Замість одного вузла:

```text
Drama
```

можна створити декілька категорій:

```text
Drama_Classic
Drama_1980s
Drama_1990s
Drama_2000s
```

або групувати за роками випуску.

У такому випадку навантаження буде розподілено між кількома вузлами, що зменшить кількість зв'язків для кожного окремого вузла та прискорить обходи графа.

Для MovieLens це особливо актуально для жанру Drama, який має понад 1600 зв'язків.


## 5.1. PageRank на графі фільмів

### Створення ребер CO_RATED

```cypher
MATCH (m1:Movie)<-[r1:RATED]-(u:User)-[r2:RATED]->(m2:Movie)
WHERE r1.rating >= 4
  AND r2.rating >= 4
  AND id(m1) < id(m2)

WITH m1, m2, count(u) AS weight

WHERE size([(m1)<-[:RATED]-() | 1]) > 20
  AND size([(m2)<-[:RATED]-() | 1]) > 20

WITH m1, m2, weight
ORDER BY weight DESC
LIMIT 50000

MERGE (m1)-[co:CO_RATED]-(m2)
SET co.weight = weight;
```

### Створення GDS-проєкції

```cypher
CALL gds.graph.project(
    'movieGraph',
    'Movie',
    {
        CO_RATED: {
            orientation: 'UNDIRECTED',
            properties: 'weight'
        }
    }
)
YIELD graphName, nodeCount, relationshipCount;
```

### Запуск PageRank

```cypher
CALL gds.pageRank.stream('movieGraph', {
    relationshipWeightProperty: 'weight'
})
YIELD nodeId, score

MATCH (m:Movie)
WHERE id(m) = nodeId

RETURN
    m.movieId AS movieId,
    m.title AS title,
    round(score * 1000) / 1000 AS pageRank

ORDER BY pageRank DESC
LIMIT 10;
```

### Результат

| Фільм | PageRank |
|---|---:|
| American Beauty (1999) | 9.659 |
| Star Wars: Episode IV - A New Hope (1977) | 9.136 |
| Star Wars: Episode V - The Empire Strikes Back (1980) | 9.100 |
| Raiders of the Lost Ark (1981) | 7.939 |
| Fargo (1996) | 7.032 |
| Silence of the Lambs, The (1991) | 6.714 |
| Godfather, The (1972) | 6.314 |
| Shawshank Redemption, The (1994) | 6.295 |
| Matrix, The (1999) | 6.288 |
| Sixth Sense, The (1999) | 6.246 |

### Що означає високий PageRank для фільму?

Високий PageRank у цьому графі означає не просто популярність фільму, а його центральність у мережі фільмів, які часто високо оцінювалися одними й тими самими користувачами.

Тобто фільм з високим PageRank не лише має багато оцінок, а й пов’язаний з іншими важливими фільмами через спільні високі оцінки користувачів.

Наприклад, `American Beauty`, `Star Wars` та `The Matrix` є не просто популярними фільмами. Вони часто входять у спільні набори улюблених фільмів багатьох користувачів, тому займають центральне місце в графі рекомендацій.


## Частина 5.2

### Чи відповідають отримані кластери інтуїтивним групам?

Частково.

Алгоритм Louvain дійсно виділив декілька великих спільнот користувачів, однак аналіз найпопулярніших жанрів показав, що більшість кластерів мають дуже схожий профіль:

- Drama
- Comedy
- Action

Лише в окремих спільнотах третє місце займає Thriller.

Тому отримані кластери не можна однозначно інтерпретувати як "любителів бойовиків", "шанувальників комедій" або "цінителів артхаусу".

Найімовірніше Louvain згрупував користувачів за схожими наборами популярних фільмів, а не за вузькими жанровими вподобаннями.


### Як перевіряли результати?

Для кожної з найбільших спільнот були знайдені найпопулярніші жанри серед фільмів, які користувачі кластера оцінили рейтингом не нижче 4.

Після цього були порівняні списки жанрів між кластерами.

Виявилося, що практично всі великі спільноти мають однакові домінуючі жанри: Drama, Comedy та Action.

Це свідчить про те, що користувачі різних кластерів мають значний перетин інтересів і дивляться багато однакових популярних фільмів.


# Частина 6. Аналіз і висновки

## 1. Graph vs SQL. Які із запитів частини 3 було б складно або неможливо написати в SQL? Чому?

Найбільша перевага графової моделі проявилася під час виконання складних запитів, які потребують багатоетапного обходу зв’язків між сутностями. Особливо це стосується запитів 5 та 6.

Запит рекомендацій «користувачі зі схожими смаками також дивилися» природно реалізується в графовій базі даних через послідовність зв’язків:

```text
User → Movie ← User → Movie
```

У Cypher такий запит описується безпосередньо через структуру графа і легко читається людиною.

У реляційній базі даних аналогічна логіка вимагала б декількох self-join до таблиці ratings, складних вкладених підзапитів та додаткових агрегацій. Зі збільшенням кількості рівнів зв’язків складність SQL-запиту швидко зростає.

Наприклад, спрощений SQL-аналог рекомендаційного запиту виглядав би приблизно так:

```sql
SELECT
    r3.movie_id,
    COUNT(DISTINCT r2.user_id) AS similar_users,
    AVG(r3.rating) AS avg_rating
FROM ratings r1
JOIN ratings r2
    ON r1.movie_id = r2.movie_id
   AND r1.user_id = 1
   AND r2.user_id <> 1
JOIN ratings r3
    ON r2.user_id = r3.user_id
LEFT JOIN ratings target_seen
    ON target_seen.user_id = 1
   AND target_seen.movie_id = r3.movie_id
WHERE r1.rating >= 4
  AND r2.rating >= 4
  AND r3.rating >= 4
  AND target_seen.movie_id IS NULL
GROUP BY r3.movie_id
ORDER BY similar_users DESC;
```

Ще складнішим є запит пошуку найкоротшого шляху між користувачами. У Neo4j для цього достатньо використати функції `shortestPath()` або алгоритми GDS. У традиційній SQL-базі даних подібні задачі потребують рекурсивних CTE або зовнішніх алгоритмів на рівні прикладної логіки.

Таким чином, графова модель значно краще підходить для задач рекомендацій, аналізу зв’язків, пошуку шляхів та роботи з мережевими структурами даних.

## 2. Де граф програє? Для яких задач із цим датасетом реляційна модель підійшла б краще?

Попри значні переваги для аналізу зв’язків, графова модель не є універсальним рішенням для всіх задач.

Для MovieLens існує багато сценаріїв, де реляційна база даних була б більш ефективною:

- побудова звітів;
- масові агрегації;
- підрахунок статистики;
- формування експорту даних;
- аналітичні запити типу OLAP.

Наприклад, обчислення середнього рейтингу для кожного фільму є типовою задачею агрегації:

```sql
SELECT
    movie_id,
    AVG(rating)
FROM ratings
GROUP BY movie_id;
```

Для таких операцій реляційні СУБД оптимізовані значно краще, ніж графові бази даних.

Аналогічно, задачі на кшталт:

- кількість оцінок по роках;
- середній рейтинг по вікових групах;
- статистика по жанрах;
- звіти для бізнес-аналітики;

часто виконуються швидше та простіше у реляційній моделі.

Графова база даних також споживає більше пам’яті при роботі з великими мережами зв’язків. Під час виконання Louvain у даній роботі виникли обмеження пам’яті, через що довелося додатково фільтрувати граф схожості користувачів.

Тому для задач звітності та масових агрегацій SQL залишається більш практичним вибором.

## 3. Покращення схеми. Які зміни в схемі прискорили б конкретні запити з частини 3?

Під час виконання роботи було виявлено декілька можливих покращень схеми графа.

### Покращення для рекомендаційного запиту

Запит рекомендацій (Запит 5) щоразу шукає користувачів зі схожими вподобаннями через спільно оцінені фільми.

Поточний шлях виглядає так:

```text
User → Movie ← User → Movie
```

Для прискорення цього сценарію можна матеріалізувати зв’язки схожості між користувачами:

```text
(User)-[:SIMILAR {weight}]->(User)
```

Такі зв’язки можуть оновлюватися періодично окремим процесом.

У цьому випадку рекомендаційний запит працюватиме значно швидше, оскільки не доведеться щоразу обчислювати схожість між користувачами.

---

### Покращення для аналізу схожості фільмів

Під час виконання PageRank довелося створювати тимчасові зв’язки:

```text
(Movie)-[:CO_RATED]->(Movie)
```

Вони формувалися через користувачів, які високо оцінили обидва фільми.

Якщо подібний аналіз виконується регулярно, доцільно зберігати такі зв’язки постійно та періодично оновлювати їх вагу.

Тоді алгоритми PageRank, Louvain та інші методи GDS працюватимуть значно швидше, оскільки не потрібно буде повторно будувати граф схожості.

---

### Покращення для супервузлів

У четвертій частині роботи було виявлено супервузли:

```text
Drama      1603 зв'язки
Comedy     1200 зв'язків
```

У великих системах кількість таких зв’язків може зрости до десятків або сотень тисяч.

Для зменшення навантаження можна застосувати підхід Node Splitting.

Наприклад, замість одного вузла:

```text
Drama
```

використовувати:

```text
Drama_Classic
Drama_1980s
Drama_1990s
Drama_2000s
```

Це дозволить рівномірніше розподілити зв’язки між вузлами та прискорити обходи графа.

## Загальний висновок

У ході виконання роботи було побудовано повноцінну рекомендаційну систему на основі графової бази даних Neo4j та датасету MovieLens.

Було реалізовано модель даних, завантажено понад мільйон оцінок користувачів, створено набір аналітичних Cypher-запитів та застосовано алгоритми Graph Data Science: PageRank, Louvain та Dijkstra.

Практична частина показала, що графова модель особливо ефективна для задач, де головну роль відіграють зв’язки між об’єктами:

- рекомендаційні системи;
- соціальні мережі;
- пошук спільнот;
- аналіз схожості;
- пошук найкоротших шляхів.

Найбільш показовими прикладами стали запити рекомендацій фільмів та пошуку найкоротших шляхів між користувачами, які в графовій моделі реалізуються природно та зрозуміло.

Водночас для класичних аналітичних задач, звітності та масових агрегацій реляційні бази даних залишаються більш ефективним інструментом.

Таким чином, графові та реляційні бази даних не є конкурентами в прямому сенсі. Кожен підхід має власну область застосування, а вибір конкретної технології залежить від характеру даних та бізнес-задач, які необхідно вирішувати.
